WBL Data Pipelines

Three layers that share a single source of truth. Canonical TR/TFR is computed once and shared everywhere. Projections build on canonical TR to project future seasons. CLI sync precomputes both for fast browser loads. All three must produce identical numbers for the same player.

Canonical Current — TR/TFR source of truth Projections — forward-looking model built on canonical TR CLI Sync — precomputes both for Supabase cache
Canonical Current — "What is this player worth right now?"

Data Sources

TrueRatingsService getPitcherTrueRatings(year) / getHitterTrueRatings(year) — authoritative TR, estimated ratings, blended rates, wOBA. Cached per year. Single source of truth for all TR values.
HitterTrueRatingsCalculationService Pure math: multi-year stat blending, scouting regression, wOBA calculation. Used by both browser and CLI (shared import).
TeamRatingsService (TFR pools) getUnifiedPitcherTfrData / getUnifiedHitterTfrData — canonical TFR with expanded pool, farm rankings.
ModalDataService resolveCanonicalBatterData / resolveCanonicalPitcherData — overrides caller-provided data with canonical TR/TFR. computeBatterProjection / computePitcherProjection — projection math using canonical blended rates.

What it does

Step 4: Compute TR Uses same HitterTrueRatingsCalculationService as browser. Results stored in player_ratings table AND reused by Step 5.5.
Step 5: Compute TFR Pitcher + hitter TFR. Stored in player_ratings + precomputed_cache.
Step 5.5: Compute Projections + Defensive Value Uses canonical Step 4 TR (not recomputed). Uses TR blended rates and wOBA directly (not re-derived from scouting ratings). League averages from prior year. PA/SB with history blending. Defensive projection from fielding scouting (raw_data.fielding) + DRS history (when available) via DefensiveProjectionService. defRuns + posAdj included in WAR. Results stored in precomputed_cache (pitcher_projections, batter_projections, defensive_lookup).
Step 6: Build Lookups Scouting, contract, DOB lookups. Name/age/stamina included for ALL players (not just draftees).

Key Design Rules (Mar 2026)

  • One TR, everywhere. TrueRatingsService is the single source of truth. CLI Step 4 computes the same TR using the same shared service. Projections use canonical TR blended rates and wOBA — never re-derived from scouting ratings.
  • Same number, every view. A player's projected WAR must be identical in the projection table, the modal WAR badge, and the modal projection line. The badge uses the projection line's computed WAR (no independent recomputation).
  • CLI = browser. sync-db.ts precomputes projections using the same canonical TR, same wOBA, same league averages (prior year), same PA formula (age curve + injury multiplier + history blending), same SB blending (projectStolenBasesWithHistory). Any drift between CLI and browser is a bug.
  • Year dropdown = target season. User selects the season they want to see (e.g., 2022). Code internally derives the stats base year (2021). The user never sees or thinks about base years.
  • Precomputed fast paths are year-aware. Cached projections only used when the requested year matches currentYear. Historical years always compute locally.
  • Custom scouting bypasses all fast paths. When hasCustomScouting is true, all precomputed Supabase caches are skipped and everything computes locally with merged my > osa scouting data.
  • Power Rankings: future years use current rosters. Selecting a year beyond the latest completed season shows players on their current teams (from players table), rated by their most recent TR.
  • Supabase teams: parentTeamId defaults to 0. WBL API stores null for MLB teams. Code uses ?? 0 to match the parentTeamId === 0 convention for root-level teams.
  • WAR includes defensive value. Batter WAR = (wRAA + replacementRuns + sbRuns + defRuns + posAdj) / runsPerWin. Defensive runs from fielding scouting ratings (position-specific formulas) + DRS when available. Positional adjustment prorated by PA. DefensiveProjectionService is the sole source. Precomputed by CLI, cached in defensive_lookup, read by browser for modal/projection/farm ranking paths.
  • Canonical prospect ranking = projected WAR. Both Farm Rankings and Top 100 badge sort by projected WAR desc (then TFR tiebreaker). One ranking everywhere.
